0) Symbols + notation cheat sheet

Foundations

Use this page like a course, reference, and optimization checklist. The formulas are intentionally practical rather than overly theoretical.

Core quantities

\(N\) = number of elements, \(b\) = bytes per element, \(B\) = bandwidth, \(T\) = time, \(F\) = floating-point operations, \(P\) = throughput, \(M\) = memory capacity.

Precision shorthand

FP64 = 8 bytes, FP32 = 4 bytes, BF16 = 2 bytes, FP16 = 2 bytes, INT8 = 1 byte, INT4 = 0.5 byte.

Useful formulas
\[ P=\frac{F}{T}, \qquad \text{Arithmetic intensity}=\frac{F}{\text{Bytes moved}}, \qquad \text{Memory footprint}=N \cdot b \] \[ \text{Speedup}=\frac{T_{\text{old}}}{T_{\text{new}}}, \qquad \text{Efficiency}=\frac{\text{speedup}}{\text{number of devices}} \]
A GPU job is fast when you keep many cores busy, move data efficiently, and avoid doing frequent small transfers between host memory and device memory.

1) What a GPU is, and when you should use one

Beginner

A GPU is a throughput-oriented processor designed to execute many similar operations in parallel. A CPU is optimized for low-latency, branch-heavy, sequential, and general-purpose tasks. For deep learning, matrix multiplication, convolutions, attention, particle simulations, and many imaging workloads, GPUs often outperform CPUs by a large margin because those tasks can be expressed as massive parallel math.

Use a GPU when

You have large tensors, batched matrix operations, repeated simulation steps, or many independent samples. Examples: training ESM, running OpenMM on long systems, image segmentation, large batched embeddings.

Do not assume a GPU helps

If your job is tiny, dominated by file I/O, Python loops, string parsing, irregular branching, or CPU-only software like many classic docking pipelines, the GPU may sit mostly idle.

Amdahl-style reminder
\[ \text{Overall speedup} \le \frac{1}{(1-f)+\frac{f}{s}} \] where \(f\) is the fraction of work accelerated and \(s\) is the speedup of that part.

If only 70% of your pipeline is GPU-accelerated and that part becomes 20× faster, the whole pipeline speedup is \[ \frac{1}{0.3+0.7/20}\approx 2.99 \] rather than 20×. This is why data loading, preprocessing, and logging matter.

2) Different types of GPUs and accelerators

Landscape

Not all accelerators are built for the same purpose. The biggest practical distinctions are consumer vs workstation vs datacenter, graphics-oriented vs compute-oriented, and GPU vs TPU/NPU.

NVIDIA ecosystem

Most mature for PyTorch/CUDA, many tutorials, strong profiler tooling, and broad support in structural biology and ML pipelines.

Strong HPC/AI push via ROCm and Instinct GPUs. Very relevant in clusters and increasingly usable for PyTorch, but compatibility still needs checking for your exact software stack.

Practical rule: for most beginner-to-intermediate scientific Python workflows, NVIDIA + CUDA is the path of least resistance. For large clusters and open-platform work, also learn ROCm and basic SYCL/OpenCL vocabulary.

3) GPU anatomy: SMs, warps, tensor cores, and VRAM

Architecture

A modern GPU is not one giant core. It is a collection of many compute units. On NVIDIA these are often discussed as Streaming Multiprocessors (SMs). Work is launched as many threads, grouped into blocks, which are then executed on SMs. Threads are scheduled in small groups called warps on NVIDIA (typically 32 threads). AMD uses analogous concepts such as compute units and wavefronts.

\[ \text{Grid} \rightarrow \text{blocks} \rightarrow \text{threads} \]

Execution view

Kernel launch → many blocks → blocks assigned to SMs → threads in each block execute cooperatively using registers and shared memory.

Data view

Tensors live in device global memory (VRAM). Fast on-chip storage includes registers and shared memory. The art of optimization is feeding compute units fast enough.

Tensor cores are specialized units that accelerate matrix-like operations, especially at reduced precision such as FP16 or BF16. They matter enormously for deep learning. A lot of modern optimization advice is really about restructuring your workload so tensor-core-friendly kernels can be used.

4) Memory hierarchy: registers, shared memory, L2, and global memory

Architecture

Most GPU performance problems are memory problems in disguise. The closer storage is to the compute unit, the faster and smaller it is.

LayerFast?Small or large?Typical use
RegistersFastestTinyPer-thread temporary values
Shared memory / local data storeVery fastSmallReuse within a block; tiling
L2 cacheFastMediumDevice-wide cache
Global memory (VRAM)Slowest on deviceLargeMain tensor storage
Host RAMMuch slower to reach from GPULargeCPU-side storage and data loading
\[ T_{\text{transfer}} \approx \frac{\text{bytes moved}}{\text{bandwidth}} \]

Numerical example. If a batch requires moving 6 GB from host to device and your effective transfer bandwidth is 12 GB/s, then the transfer alone costs roughly \[ T \approx \frac{6}{12}=0.5 \text{ s}. \] If compute itself is only 0.2 s, data transfer is your bottleneck.

This is why pinned memory, prefetching, asynchronous copies, larger batches, and keeping tensors on-device across multiple steps are often more valuable than fancy kernel tweaks.

5) Throughput, bandwidth, arithmetic intensity, and the roofline idea

Performance model

A simple performance model helps you reason before you profile. Many kernels are either: memory bound (limited by bytes moved) or compute bound (limited by available arithmetic throughput).

\[ \text{Arithmetic intensity} = \frac{\text{FLOPs}}{\text{Bytes moved}} \] \[ T_{\text{compute}} \approx \frac{\text{FLOPs}}{\text{Peak FLOP/s}}, \qquad T_{\text{memory}} \approx \frac{\text{Bytes}}{\text{Bandwidth}} \]

The rough runtime lower bound is controlled by the slower side. If memory time dominates, optimize memory movement. If compute time dominates, optimize math kernels, precision, and tensor-core usage.

Numerical example. Suppose a kernel does \(2\times10^{12}\) FLOPs and moves \(2\times10^{11}\) bytes. Its arithmetic intensity is \[ \frac{2\times10^{12}}{2\times10^{11}}=10 \text{ FLOPs/byte}. \] If your effective memory roof corresponds to 6 FLOPs/byte for that device, the kernel is likely memory-limited.

6) Precision types: FP64, FP32, TF32, BF16, FP16, INT8, INT4

Numerics

Precision affects speed, memory, and sometimes numerical stability. In practice: FP32 is the classic default, BF16 and FP16 are common for training, and INT8/INT4 are common for inference or quantized training research.

FP162Training and inferenceCan underflow/overflow more easily; loss scaling often used

\[ \text{Memory reduction from FP32 to BF16} = \frac{4-2}{4}=50\% \]

Numerical example. A 7-billion-parameter model stored as FP32 weights needs about \[ 7\times10^9 \cdot 4 \approx 28 \text{ GB} \] just for weights. In BF16 that becomes about 14 GB. That difference can decide whether the model fits.

7) Software stacks: CUDA, ROCm, SYCL, OpenCL, PyTorch, JAX, TensorFlow

Software stack

A GPU is only useful if your software stack can talk to it.

CUDA

NVIDIA’s programming platform and ecosystem. Most scientific deep learning tutorials, extension libraries, and profiling tools are CUDA-first.

ROCm / HIP

AMD’s open software stack for GPU compute. Increasingly important in HPC and large clusters, with PyTorch and other framework support for many workloads.

SYCL / oneAPI

A cross-platform C++ model for heterogeneous programming. Important if you care about portability beyond a single vendor.

OpenCL

A long-standing open standard for heterogeneous compute. Less dominant in ML tutorials than CUDA, but still conceptually important.

At the framework level, PyTorch, JAX, and TensorFlow hide most low-level kernel management. That is why many scientists can be productive without writing CUDA kernels directly. But to optimize well, you still need to understand the hardware model underneath.

Beginner recommendation: learn PyTorch device management + profiling + mixed precision first. Then learn how those concepts map to lower-level CUDA/HIP ideas.

9) First principles in PyTorch: tensors, devices, synchronization, and timing

Hands-on

In PyTorch, you move data and models to a device. Timing must be done carefully because GPU launches are asynchronous. If you time without synchronization, you may benchmark the queueing of work rather than the work itself.

\[ \text{Measured time} \approx \text{launch overhead only} \] if you forget to synchronize around a GPU benchmark.
Annotated Python: correct CPU vs GPU matrix multiplication benchmark

Numerical example. If a GPU kernel launch returns in 0.0003 s but the synchronized end-to-end operation takes 0.045 s, the true kernel time is the latter. This is one of the most common beginner mistakes.

10) Estimating VRAM before you crash: parameters, gradients, optimizer states, activations

Planning

The easiest way to waste time on a GPU cluster is to launch a job that cannot fit. Estimate memory before training.

\[ \text{Weights memory} \approx N_{\theta}\cdot b \] \[ \text{Training memory} \approx \text{weights}+\text{gradients}+\text{optimizer states}+\text{activations} \]

For Adam-like optimizers, optimizer states can be roughly 2× parameter storage, sometimes more depending on implementation and master copies. This is why training usually needs much more memory than inference.

Numerical example. Suppose a model has 800 million parameters in BF16: \[ 800\times10^6 \cdot 2 \approx 1.6 \text{ GB} \] for the raw weights. Gradients add about another 1.6 GB. Adam states can add roughly 3.2 GB more. Before counting activations, you are already near 6.4 GB.

11) Kernel launches, occupancy, and why tiny workloads underuse a GPU

Optimization

A kernel is device code launched across many threads. Good kernels expose enough parallel work to keep many SMs busy. Bad kernels are too small, too branchy, or too memory-stalled.

\[ \text{Occupancy} \approx \frac{\text{active warps on an SM}}{\text{maximum supported warps on that SM}} \]

Occupancy is not the same as performance, but low occupancy can be a symptom of limited parallelism, excessive register use, or block-size choices that prevent good scheduling.

Numerical example. If an SM can host 64 warps but your kernel configuration allows only 24 active warps, then \[ \text{occupancy}\approx \frac{24}{64}=37.5\%. \] That does not automatically mean the kernel is bad, but it is a sign to inspect launch configuration and register pressure.

Scientists often encounter this when processing one sequence at a time, one image at a time, or very small matrices. Batching is frequently the simplest fix.

12) Matrix multiplication: the central GPU workload

Core workload

If you understand matrix multiplication, you understand much of modern GPU computing. Linear layers, attention projections, embeddings, many graph operations, and even some simulation kernels reduce to GEMM-like patterns.

\[ C_{m\times n}=A_{m\times k}B_{k\times n} \] requires roughly \[ 2mnk \] floating-point operations.

Numerical example. For \(m=n=k=4096\), \[ 2\cdot 4096^3 \approx 1.37\times10^{11}\ \text{FLOPs}. \] If that multiply takes 0.01 s, the achieved throughput is roughly \[ 1.37\times10^{13}\ \text{FLOP/s}=13.7\ \text{TFLOP/s}. \]

Annotated Python: measure GEMM throughput and report effective TFLOP/s
Many optimization tricks—mixed precision, kernel fusion, better batching, and layout tuning—ultimately exist to make your code behave more like large efficient matrix multiplications.

13) Convolution: why CNN workloads behave differently from plain GEMM

Deep learning

Convolution reuses small kernels across spatial positions. Frameworks often lower convolutions to matrix multiplication internally or use specialized kernels.

\[ H_{\text{out}}=\left\lfloor \frac{H+2P-K}{S}\right\rfloor +1,\qquad W_{\text{out}}=\left\lfloor \frac{W+2P-K}{S}\right\rfloor +1 \]

Numerical example. Input \(224\times224\), kernel \(3\times3\), stride \(1\), padding \(1\): \[ H_{\text{out}}=W_{\text{out}}=224. \] With stride \(2\), the spatial map roughly halves.

Annotated Python: compute convolution output shapes and activation sizes
For microscopy and medical imaging, activation memory often dominates. Large feature maps are one reason segmentation models can exhaust VRAM quickly even when parameter counts look modest.

14) Profiling before guessing: PyTorch profiler, kernel traces, and bottleneck hunting

Profiling

Never optimize blind. The right sequence is: measure → identify bottleneck → change one thing → re-measure.

What to profile

Operator times, kernel traces, host-to-device copies, memory peaks, GPU utilization, data-loader stalls, and synchronization hotspots.

Typical surprises

Tiny kernels launched too often, excessive .cpu()/.numpy() calls, augmentation on the critical path, and hidden synchronization.

Annotated Python: collect a basic PyTorch profile
A strong workflow is: first use framework-level profiling, then use device-level tools if needed. For many scientists, framework profiling already explains most practical slowdowns.

15) Memory optimization patterns that matter in practice

Optimization

When GPU jobs fail or run slowly, these are the highest-yield levers:

  1. Use a larger but still safe batch size to better saturate the device.
  2. Reduce precision where appropriate.
  3. Use gradient checkpointing to trade compute for memory.
  4. Fuse operations or use compiled paths so fewer kernels move data back and forth.
  5. Avoid repeated host-device transfers.
  6. Use pinned memory and non-blocking transfers in the input pipeline.
\[ \text{Activation checkpointing: memory}\downarrow,\quad \text{extra recomputation}\uparrow \]

Numerical example. If checkpointing cuts activation storage from 10 GB to 4 GB but adds 20% more compute, it may be worth it if the alternative is an out-of-memory failure.

16) Mixed precision training, loss scaling, and stable fast training

Training optimization

Automatic mixed precision (AMP) lets frameworks use lower precision for many operations while preserving accuracy where needed. This often improves speed and reduces memory.

\[ \tilde{L}=sL \]

In FP16 training, a scale factor \(s\) may be used so tiny gradients do not underflow during backpropagation. BF16 often needs less babysitting because it keeps a wider exponent range.

Annotated PyTorch: one training step with AMP
In modern practice, BF16 is often the most comfortable reduced-precision choice on supported hardware. But you still need validation checks because “faster” is useless if the model silently degrades.

17) Gradient accumulation, checkpointing, and fitting bigger models than your VRAM allows

Training optimization

If the full batch does not fit, split it into microbatches, backpropagate each, and update after accumulating gradients.

\[ \text{Effective batch size}= \text{microbatch size}\times \text{accumulation steps}\times \text{number of devices} \]

Numerical example. If you can fit only 2 samples per GPU, use 8 accumulation steps on 4 GPUs: \[ 2\times 8\times 4=64 \] effective batch size.

Annotated PyTorch: gradient accumulation loop
This is extremely useful for protein models, long-sequence transformers, and 3D imaging workloads.

18) Data pipeline optimization: the GPU is only as fast as the input pipeline

Pipeline

One of the most common real-world bottlenecks is not the model—it is the input pipeline. If the GPU waits for the next batch, utilization drops even if the model is efficient.

19) Multi-GPU and distributed training: DDP, sharding, and communication costs

Advanced scaling

Once one GPU is full or too slow, you scale out. The simplest scaling pattern is data parallelism: each GPU processes a different batch slice, then gradients are synchronized.

\[ T_{\text{step}} \approx T_{\text{forward/backward}} + T_{\text{communication}} \]

Communication is why scaling is never perfectly linear. Interconnects such as NVLink and topology choices matter. If communication dominates, more GPUs may yield disappointing gains.

DDP

Best first choice when the model fits on one GPU and you want to scale training across several GPUs.

FSDP / sharded approaches

Useful when the model itself is too large for one GPU. Parameters, gradients, and optimizer states are sharded across devices.

Minimal DDP launch template
If your model fits on one GPU, fix single-GPU efficiency first. Scaling a bad one-GPU setup to eight GPUs usually just wastes money faster.

20) Inference optimization: batch sizing, compilation, quantization, and serving

Inference

Training optimization and inference optimization are related but not identical. For inference you often care about: latency, throughput, memory footprint, and cost per request.

\[ \text{Latency} = \text{time per request}, \qquad \text{Throughput} = \text{requests or samples per second} \]

Higher batch sizes usually improve throughput but worsen latency. Quantization and compiled graphs can improve both, depending on the workload.

22) End-to-end protein-ML GPU pipeline: from FASTA to embeddings to fine-tuning

Pipeline

Here is a realistic GPU-aware pipeline for a protein sequence project:

  1. Read FASTA and metadata on CPU.
  2. Bucket sequences by length to reduce padding waste.
  3. Move padded batches to GPU.
  4. Extract embeddings or train a classifier with AMP.
  5. Profile sequence-length distribution and batch-size scaling.
  6. Save embeddings or checkpoints asynchronously when possible.
\[ \text{Padding waste} = 1-\frac{\sum_i L_i}{B\cdot L_{\max}} \]

Numerical example. Suppose a batch has sequence lengths \([120,118,117,30]\). If you pad to \(L_{\max}=120\) with batch size \(B=4\), total allocated positions are \(4\cdot120=480\), while useful positions are \(385\). Waste is \[ 1-\frac{385}{480}\approx 19.8\%. \] Bucketing by length reduces this waste and often increases throughput.

23) End-to-end MD GPU pipeline: setup, simulate, monitor, and optimize

Pipeline

A practical GPU-first MD pipeline might look like:

  1. Prepare structure and force field files on CPU.
  2. Build the system and choose integrator settings.
  3. Select CUDA or HIP platform.
  4. Benchmark a short equilibration run.
  5. Record ns/day or steps/s, not just wall-clock for one step.
  6. Tune output frequency, precision mode, and system setup choices.
Writing trajectories too often can dominate runtime. Performance tuning is not only about kernels; it is also about not doing unnecessary I/O.

26) Benchmarking playbook: how to compare devices honestly

Methodology

A good GPU benchmark is:

  1. Representative of your real workload.
  2. Warm-started so initialization overhead does not dominate.
  3. Synchronized for accurate timing.
  4. Repeated several times.
  5. Reported with both performance and memory numbers.
\[ \text{Median speedup} = \frac{\text{median CPU time}}{\text{median accelerator time}} \]

Numerical example. Suppose step times on a GPU are 0.42, 0.40, 0.41, 0.95, 0.40 s. The median is 0.41 s and is more trustworthy than the mean because the 0.95 s run may reflect a transient stall.

27) Hands-on mini projects and exercises

Practice

Use these as checkpoints to turn the course into skill:

  1. Benchmark CPU vs GPU matrix multiplication for several sizes and plot where the GPU becomes worthwhile.
  2. Profile a small CNN and identify whether it is data-loader-limited or compute-limited.
  3. Fit a transformer batch by combining AMP, length bucketing, and gradient accumulation.
  4. Compare two cloud runtimes using cost per processed sequence rather than just raw speed.
  5. Take one of your own protein or imaging pipelines and write down the time split between I/O, preprocessing, model compute, and post-processing.
If you can explain why a benchmark changed—not just that it changed—you are moving from beginner to advanced GPU usage.
